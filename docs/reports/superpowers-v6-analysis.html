Plugin Strategy Analysis · June 2026

What Superpowers v6 Teaches Compound Engineering

Obra's Superpowers 6.0 is, underneath the changelog, a cost-and-fidelity campaign on an agentic development methodology. This report dissects what actually changed, maps each idea onto our compound-engineering plugin, and turns the overlap into a prioritized set of changes we can make.

⚡ The one-paragraph version

Superpowers v6 didn't add features — it made its existing methodology measurably cheaper without losing quality (their evals report ~2× faster, ~50% fewer tokens). It did that with seven moves: merging two task reviewers into one dual-verdict pass, handing diffs and task text between agents as files instead of pasted text, a "cannot verify from the diff" escape hatch, explicit per-dispatch model tiering, a durable progress ledger that survives context compaction, plans that carry Global Constraints and per-task Interfaces blocks, and — the real unlock — a cross-harness eval harness ("drill") that runs real agent sessions and grades skill behavior.

  1. We're already ahead on several fronts — model tiering, harness-agnostic vocabulary, right-sizing, and TDD-with-evidence all exist in our plugin. This is convergent evolution, not a gap.
  2. Five mechanisms are cleanly borrowable and high-leverage: the ⚠ "cannot-verify" verdict, file-handoff for review packets, a durable progress ledger, and Global-Constraints / Interfaces blocks in plans.
  3. The strategic gap is measurement. We make a deliberate, expensive quality bet (6–9 persona reviewers per review) but we have no standing way to measure whether it's worth the cost. Superpowers' whole v6 was possible because they built the eval + cost harness first. That's the thing to copy.
Part 1

What actually changed in v6

The headline number — "twice as fast, ~50% fewer tokens, same quality" — is an outcome. The interesting part is the seven mechanisms that produced it. Superpowers' spine is subagent-driven-development (SDD): a controller agent dispatches one implementer per task, then a reviewer per task, then a whole-branch review at the end. Every v6 change tightens that loop.

2→1
Reviewers per task. Spec-compliance and code-quality merged into one dual-verdict pass.
~41%
Reviewer output reduction from a "terse contract": every line is a verdict, a file:line finding, or a check actually run.
~50%
Fewer tokens end-to-end at parity quality, per their N=5 eval gates.

Part 2

Convergent evolution — and where we made a different bet

The most important framing for this whole report: our plugin and Superpowers are two independent solutions to the same problem, and on most axes they've converged. Reading v6 as "things we're missing" would be wrong. The useful reading is "where did they validate a choice we already made, where can we cheaply borrow, and where did they deliberately go a different direction than us — and why."

Where we're already strong

Our ce-code-review already tiers models (Haiku for the trivial-PR gate, a mid-tier for most personas, the session model for correctness/security/adversarial). Our skills already speak in actions ("dispatch a subagent") and resolve the right per-harness tool for Claude Code, Codex, Gemini, and Pi. ce-plan already right-sizes and our reviewers already demand evidence. We did not need v6 to tell us these are correct — but v6 independently validates them.

Where the different bet costs us

We fan out to 6–9 specialized persona reviewers per review and dedup their findings. That buys real coverage — Superpowers' single reviewer cannot match a dedicated ce-data-migration-reviewer or ce-julik-frontend-races-reviewer. But it is expensive, and we have never measured the coverage-per-dollar. Superpowers' two-tier shape (cheap per-task gate during iteration, thorough review once at the end) is a hybrid we should seriously consider.

Part 3

The review rewrite, side by side

This is the heart of v6 and the place our designs most directly disagree. Worth looking at both honestly.

Three things from their reviewer we should take regardless of the breadth debate

What Superpowers does

⚠ "Cannot verify from the diff." If a requirement lives in unchanged code or spans tasks, the reviewer reports it as a ⚠ line for the controller to check — "instead of broadening your search."

What we do today

Our ce-coherence-reviewer does the opposite: findings it "cannot verify" are suppressed entirely (below confidence anchor 50). The thing we can't check silently disappears instead of becoming a human to-do.

What Superpowers does

Diff travels as a file. One script bakes git log + --stat + diff -U10 into a packet on disk; the reviewer reads the path once and is told the context lines are the changed files — don't re-open them.

What we do today

We compute the diff, then paste it inline into each of 6–9 reviewer prompts. The same diff body is duplicated N times and stays resident. We already write reviewer outputs to a run dir — inputs could go the same way.

What Superpowers does

Rationale never downgrades severity. "Left it per YAGNI" is treated as the implementer grading their own work; a plan-mandated defect is still flagged (labeled plan-mandated) for the human to decide.

What we do today

We have read-only validators and confidence gating, which is good. But we don't have an explicit rule that a stated design rationale in the diff/PR can't suppress a finding — a cheap, sharp addition to our reviewer contract.

Part 4

Plans as contracts: constraints & interfaces

Superpowers' measured finding is blunt and worth internalizing: when a plan carries complete structure — exact constraints, exact cross-task interfaces, right-sized tasks — the implementation body becomes "transcription plus testing." The dollars and the bugs both come from the controller and from interface re-derivation, not from writing function bodies. Their v6 plan template encodes this with two blocks we don't have as first-class fields.

Global Constraints block

A verbatim plan-header section: version floors, dependency limits, naming/copy rules, platform requirements — "every task's requirements implicitly include this section." A go 1.26.1 floor once shipped violated because it lived in prose and no one pasted it downstream. A fixed heading makes it mechanically extractable into every task brief.

Per-task Interfaces block

Consumes: what this task uses from earlier tasks (exact signatures). Produces: exact names/types later tasks rely on. "An implementer who sees only its own task still knows its neighbors' contracts." This is the single line that removes the implementer's ~13-call rediscovery of context.

Our current state

Our ce-plan is genuinely strong here — Implementation Units carry Files:, per-unit test scenarios, and we already reclassify plan depth upward when work touches "external contract surfaces" (exported APIs, shared types). What we lack is (a) a dedicated verbatim project-wide constraints block that's guaranteed to reach every implementer, and (b) consumes/produces interfaces as a named per-unit field rather than something the implementer infers from the Files list and prose. Both are small additions to references/plan-sections.md, and both directly serve our own ce-work executor.

Part 5 — the strategic one

The measurement gap is the real lesson

Everything above is borrowable tactics. This is the part that changes how we work. Superpowers could ship a "50% cheaper, same quality" release only because they built the instrument first: drill, an eval harness that launches a real agent session in tmux, has an LLM actor play the user, runs the skill to completion, and has an LLM verifier grade whether the skill triggered and was followed — across Claude Code, Codex, Gemini, Pi, and Antigravity. They ran 25+ cost experiments with N=5 quality gates and a blind A/B parity check, and killed the ones that regressed quality (a cheap reviewer tier "flagged 0 of 10 planted defects" — so it died at the gate).

The honest comparison

Our plugin tests skill behavior by hand — AGENTS.md tells contributors to use skill-creator to spawn a subagent and eyeball the result. We have exactly one seed of automation: ce-sessions/evals/ (an evals.json + a grader.md). We also have ce-optimize, which already knows how to run parallel experiments with hard gates and LLM-as-judge scoring — we have most of the machinery, just pointed at user problems instead of at our own skills. The gap is not capability; it's that we've never turned the instrument on ourselves.

Until we can measure cost-and-quality of our own workflows, every efficiency decision we make (Is the 9-reviewer fan-out worth it? Should the per-task gate be one cheap reviewer? Does pasting diffs inline actually cost us much?) is an argument, not a result. Building that harness is the highest-leverage thing on this list, because it's what makes everything else on this list decidable.

Part 6

Recommendations, prioritized

Ordered by leverage-to-effort. Each is grounded in a specific v6 mechanism and a specific place in our plugin.

R1 Add a "cannot verify from the diff" channel to code reviewhigh impactlow effort

Our reviewers suppress what they can't verify; v6 surfaces it. Surfacing is strictly safer — an un-checkable requirement becomes a human to-do instead of a silent gap.

v6 source
subagent-driven-development/task-reviewer-prompt.md — the ⚠ line reported alongside ✓/✗.
Change
Add an unverifiable[] field to the reviewer JSON schema and a "Cannot verify from this diff" section to the ce-code-review report. Update ce-coherence-reviewer (and the shared reviewer template) so "cannot verify" routes to that channel instead of the suppress-below-50 path.
Files
skills/ce-code-review/SKILL.md, agents/ce-coherence-reviewer.md, shared reviewer template.

R2 A durable progress ledger for ce-workhigh impactlow–med effort

v6 calls re-dispatching an already-finished task sequence after compaction "the single most expensive failure observed." We track progress in the in-session task list (dies on compaction) and git history (must be re-derived). A named ledger makes recovery deterministic.

v6 source
subagent-driven-development/SKILL.md — <git-dir>/sdd/progress.md, "trust ledger over memory."
Change
Have ce-work append U<n>: complete (commits <a>..<b>, review clean) to a per-run ledger as each Implementation Unit lands, and read it on resume to skip finished units. Fits our scratch convention: cross-invocation reusable → /tmp/compound-engineering/ce-work/<run-id>/progress.md (or .context/ if we want it user-inspectable across a checkout).
Files
skills/ce-work/SKILL.md + a short reference for the ledger format.

R3 Add Global-Constraints & per-unit Interfaces to planshigh impactlow–med effort

v6's measured result: structure and interfaces carry fidelity, bodies are transcription. We already reclassify on contract surfaces; making constraints and consumes/produces first-class fields closes the loop into ce-work.

v6 source
writing-plans/SKILL.md — ## Global Constraints header + per-task Interfaces: consumes/produces.
Change
Add an optional verbatim Global Constraints section to our plan hard-floor/include-when-material catalog, and an Interfaces (consumes / produces) field to Implementation Units when units have cross-unit contracts. Teach ce-work to pass both into per-unit execution.

R4 Build a skill-behavior eval harness (the big bet)highest impacthigh effort

This is what made v6 possible. Without it, our efficiency choices stay arguments. We already have the pieces: ce-optimize runs gated parallel experiments with LLM-judge scoring, and ce-sessions/evals/ is a working seed.

v6 source
"drill" + evals/ submodule (docs/testing.md): tmux session · LLM actor · LLM verifier · YAML scenarios with planted-defect assertions · per-harness backends.

Generalize ce-sessions/evals into a repo-level evals/ harness with YAML scenarios ("reviewer catches the planted bug," "plan includes interface for cross-unit type," "ce-work resumes from ledger after compaction"), an actor + verifier, and cost capture per run. Start with 3–5 scenarios for our highest-stakes skills; keep it manual-only (not CI) like theirs.
Files
New evals/ tree; reuse ce-optimize scoring patterns; lean on the skill-creator dispatch pattern AGENTS.md already endorses.

R7 Audit every subagent dispatch for explicit model + terse returnmed impactlow effort

We tier models well in code review, but other skills that dispatch subagents (research in ce-plan deepening, ce-compound, brainstorm fan-outs) may silently inherit the expensive session model. v6's guard: always state model:; and "turn count beats token price" (cheapest model that one-shots, not unconditionally cheapest).

R8 Consider a two-tier review: cheap per-task gate, full fleet at PRmed impactmed effortneeds R4 first

v6's shape — one cheap reviewer gating each task during iteration, one thorough review at the end — maps onto our world as: a single merged dual-verdict reviewer inside ce-work's loop, reserving the 6–9-persona fleet for the PR gate. But this trades coverage for cost, so do not ship it on intuition — gate it on R4's measurements.

Change
